# Export-as-Input Refactoring Audit — S2O

## Purpose

S2O captures the follow-up work needed to remove remaining local export-as-input workflows from the project.

The project rule is intentionally strict:

> Generated files may be human-readable review artifacts, but they must not become hidden pipeline inputs, activation gates, destructive-operation inputs, migration dependencies or production/cloud execution dependencies.

This exists because CSV/Excel-style engineering tends to become permanent once it is allowed into operational flows. The project should avoid that debt before cloud migration and productionization.

## Current Boundary

S2L and S2N for Finanz Informatik have already been refactored to use live connector-candidate preview data and current database evidence. Their Markdown/JSON outputs are review artifacts only.

S2O does not reopen that decision. It tracks remaining legacy workflows that still need to be refactored or retired.

### Historical burden hot-store removal

Observed files:

- `scripts/prepare_historical_burden_hot_store_removal.py`
- `scripts/remove_historical_burden_from_hot_store.py`
- `tests/test_historical_burden_hot_store_guarded_removal.py`
- `tests/test_historical_burden_hot_store_removal.py`

Current status:

- Stage 1 has introduced DB-backed proposed review batches and review items.
- `scripts.prepare_historical_burden_hot_store_removal` now creates review state in the database and writes Markdown/JSON as human-readable outputs only.
- Stage 2 is still open: `scripts.remove_historical_burden_from_hot_store` must be refactored to read an approved DB batch instead of a local manifest/candidate file.

Target state:

- The removal operation should read explicitly reviewed database state, not a local CSV file path.
- Human-readable exports may still document what happened, but they must not drive the operation.

## Refactoring Sequence

1. Classify all remaining export-as-input occurrences as historical spike, review artifact or operational blocker.
2. Retire the Finanz Informatik legacy S2J/S2K local handoff. Done in S2O-A1.
3. Design and implement a DB-backed historical-burden removal review state. Stage 1 done: proposed DB batches/items exist.
4. Add a regression check or documented review command that catches new export-as-input patterns before they enter production-facing code.
5. Update source-analysis and roadmap documentation after each refactor step.

## Acceptance Criteria

S2O is complete when:

- no active connector, gate, review, cleanup or removal workflow depends on a local CSV/Excel-style file generated by a previous step
- destructive operations use database-backed reviewed state instead of local file handoffs
- local exports are only human-readable outputs or archived spike evidence
- cloud/CI execution does not require manually copied export files
- documentation clearly distinguishes review artifacts from durable pipeline inputs

## Non-Goal

S2O does not forbid all generated files. Markdown and JSON outputs remain useful for human review, audit trails and local inspection.

The boundary is simple: generated files may explain decisions, but they must not make decisions.
